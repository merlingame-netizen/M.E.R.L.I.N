M.E.R.L.I.N. — LoRA Training Specification

Fine-tuning Qwen 2.5-1.5B pour la narration celtique — QLoRA 4-bit
Derniere mise a jour : 24 fevrier 2026

Objectif de ce document Specification detaillee du fine-tuning LoRA pour ameliorer 5 competences specifiques du modele Qwen 2.5-1.5B dans le contexte de M.E.R.L.I.N. Ce document couvre l'architecture adapter, le pipeline complet avec scripts, et les criteres GO/NO-GO.

Objectif

Ameliorer 5 competences specifiques du Qwen 2.5-1.5B pour M.E.R.L.I.N., sans degrader les capacites generales du modele. Le LoRA cible exclusivement les taches ou le prompt engineering seul ne suffit pas.

Principe fondamental Le LoRA ne remplace PAS le prompt engineering. Il le complete pour les cas ou le modele base echoue malgre des prompts optimaux. Voir la section «Ce que LoRA NE DOIT PAS faire».

Competences Ciblees

Ce que LoRA DOIT ameliorer (5 competences)
#CompetenceNb ExamplesJustificationDataset Source
L1 Ton celtique Merlin
Vocabulaire druidique, poetique, referents culturels bretons		 200 Le modele base produit du francais generique. Le vocabulaire specifique (dolmen, nemeton, ogham, aubepine) n'apparait que si explicitement injecte dans le prompt. Gold examples manuels + augmentation GPT-4
L2 Format [NARRATION]/[CHOIX]
Structure stricte avec delimiteurs		 150 Le modele base oublie les delimiteurs ~15% du temps. Un fine-tuning sur la structure reduit le taux d'erreur de parsing a <2%. Cartes generees filtrées (100% conformes) + synthese
L3 Choix significatifs (verbes precis)
Actions concretes, pas de choix generiques		 180 "Avancer" / "Rester" / "Partir" reviennent trop souvent. Les choix doivent etre des verbes specifiques lies au contexte (fouiller, invoquer, ecouter, trancher). Gold examples + corrections manuelles des sorties faibles
L4 Personnalite coherente
Merlin garde son ton entre les cartes		 120 Sur 10+ cartes, le modele derive vers un ton neutre. Le LoRA ancre la personnalite (humour noir, sagesse cryptique, malice). Sequences de 5-10 cartes avec continuity scoring
L5 Conscience des 3 etats Triade
Narration adaptee a Corps/Ame/Monde		 100 Le modele ignore souvent les etats fournis. Un "Corps Epuise" devrait generer des descriptions de fatigue physique, pas des scenes heroiques. Paires etat-narration annotees + tests de regression

Total dataset : ~750 examples (ChatML format). Apres augmentation et deduplication : ~1000-1200 examples.

Ce que LoRA NE DOIT PAS faire (6 exclusions)
#Capacite exclueMethode actuellePourquoi PAS LoRA
P1 Adaptation biome (vocabulaire) Injection {biome_words} dans le prompt Le prompt injecte deja les mots cles. Le LoRA sur-specialiserait le modele a un sous-ensemble de biomes.
P2 Conscience temporelle (saison, jour) Variable {season} + {day}/{max} Le modele comprend deja les saisons via le prompt. Pas de gain mesurable.
C1 Equilibrage des effets (GM) GBNF grammar + validation code Les effets sont des valeurs numeriques. La grammaire GBNF + validation merlin_effect_engine.gd garantissent 99.5% de conformite.
C2 Detection danger / guerison Logique dans merlin_card_system.gd Le code detecte quand un aspect est «Bas» et injecte la contrainte dans le prompt GM. Pas besoin d'apprendre ca au modele.
C3 Economie du Souffle Logique dans merlin_store.gd Le systeme de Souffle est entierement gere par le code. Le LLM n'a pas besoin de «comprendre» l'economie.
C4 Cross-run references RAG v2.0 dans rag_manager.gd Les references cross-run sont injectees via le systeme RAG. Le LoRA ne peut pas memoriser les runs precedents.
Regle de decision Si le prompt engineering OU le code peuvent resoudre le probleme → pas de LoRA.
LoRA uniquement quand le modele echoue malgre un prompt optimal.

Pipeline Complet (7 etapes)

1

Preparer le dataset

Assembler les examples ChatML dans un fichier JSONL. Chaque ligne = un example avec system, user, assistant.

# Format ChatML JSONL (une ligne par example)
{"messages": [
  {"role": "system", "content": "Tu es Merlin, druide FOU..."},
  {"role": "user", "content": "Jour 5/30. Saison: Beltane..."},
  {"role": "assistant", "content": "[NARRATION]\nLes flammes..."}
]}

# Fichier de sortie
data/ai/training/merlin_lora_v8.jsonl

Sources : gold examples manuels, sorties LLM filtrées (score > 8/10), augmentation GPT-4 pour diversite.

2

Valider le dataset

Verifier la conformite (format, longueur, doublons, qualite).

python tools/lora/validate_dataset.py \
  --input data/ai/training/merlin_lora_v8.jsonl \
  --check-format chatml \
  --check-duplicates \
  --min-length 50 \
  --max-length 512 \
  --check-french-ratio 0.8 \
  --report data/ai/training/validation_report.json

Criteres : 0 doublons, 100% format valide, 80%+ francais, longueur moyenne 200-350 tokens.

3

Entrainer le LoRA

Lancer l'entrainement QLoRA avec tous les hyperparametres.

python tools/lora/train_narrator_lora.py \
  --model_name "Qwen/Qwen2.5-1.5B" \
  --dataset data/ai/training/merlin_lora_v8.jsonl \
  --output_dir data/ai/adapters/narrator_v8/ \
  --lora_r 16 \
  --lora_alpha 32 \
  --lora_dropout 0.05 \
  --target_modules "q_proj,k_proj,v_proj,o_proj,gate_proj,up_proj,down_proj" \
  --learning_rate 2e-4 \
  --warmup_ratio 0.1 \
  --per_device_train_batch_size 4 \
  --gradient_accumulation_steps 4 \
  --num_train_epochs 3 \
  --max_seq_length 512 \
  --bf16 \
  --logging_steps 10 \
  --save_strategy "epoch" \
  --seed 42
4

Convertir en GGUF

Merger l'adapter avec le modele base et convertir au format GGUF pour Ollama.

python tools/lora/convert_to_gguf.py \
  --base_model "Qwen/Qwen2.5-1.5B" \
  --adapter_path data/ai/adapters/narrator_v8/ \
  --output_path addons/merlin_llm/adapters/merlin-narrator-v8.gguf \
  --quantization Q4_K_M

Taille attendue : ~1.0 GB (modele merge) ou ~12 MB (adapter seul, si Ollama supporte le chargement separe).

5

Creer Modelfile Ollama

Creer le fichier de configuration Ollama pour le modele fine-tune.

# Modelfile
FROM qwen2.5:1.5b
ADAPTER ./addons/merlin_llm/adapters/merlin-narrator-v8.gguf

PARAMETER temperature 0.75
PARAMETER top_p 0.92
PARAMETER top_k 40
PARAMETER repeat_penalty 1.35
PARAMETER num_predict 150

TEMPLATE """<|im_start|>system
{{ .System }}<|im_end|>
<|im_start|>user
{{ .Prompt }}<|im_end|>
<|im_start|>assistant
"""

SYSTEM "Tu es Merlin, druide FOU de Broceliande."
6

Benchmark (50 runs)

Comparer modele base vs modele LoRA sur 50 prompts standardises.

python tools/lora/benchmark_lora.py \
  --base_model "qwen2.5:1.5b" \
  --lora_model "merlin-narrator" \
  --prompts data/ai/training/benchmark_prompts.jsonl \
  --num_runs 50 \
  --metrics "french_ratio,format_compliance,jaccard_diversity,celtic_vocab,latency_p50,latency_p90,personality_consistency" \
  --output data/ai/training/benchmark_results.json \
  --go_nogo_thresholds data/ai/training/thresholds.json
7

Deployer

Creer le modele Ollama et mettre a jour le code.

# Creer le modele dans Ollama
ollama create merlin-narrator -f Modelfile

# Tester
ollama run merlin-narrator "Jour 1/30. Biome: Broceliande."

# Mettre a jour merlin_ai.gd
# Changer: model_name = "merlin-narrator" (au lieu de "qwen2.5:1.5b")
# Pour le Narrator uniquement. Le GM reste sur qwen2.5:1.5b.

Benchmarks GO/NO-GO

Criteres bloquants Si UNE metrique est en dessous du seuil GO, le deploiement est REFUSE. Corriger le dataset ou les hyperparametres et relancer.
MetriqueBase (Qwen 2.5-1.5B)Seuil GOMethode de mesure
French ratio 80% ≥ 85% Detecteur langue (langdetect) sur output complet
Format compliance 85% ≥ 95% Regex parser [NARRATION]+[CHOIX]+A/B/C
Jaccard diversity < 0.7 < 0.65 Score Jaccard moyen entre toutes paires de 50 sorties
Celtic vocab density ~0.5 mots/carte ≥ 1.5 mots/carte Compteur de mots celtiques (liste de 50 termes) par sortie
Latence p50 (warm) ~7s < 10s Chrono sur 50 appels Ollama (warm cache)
Latence p90 (warm) ~11s < 15s Percentile 90 sur 50 appels
Personality consistency ~60% ≥ 80% Score humain (5 reviewers) sur coherence Merlin dans 10 sequences de 5 cartes
