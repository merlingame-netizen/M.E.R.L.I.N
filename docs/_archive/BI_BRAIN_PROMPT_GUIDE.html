Principes de Prompt Engineering pour 1.5B

Regle d'or : un modele 1.5B n'est PAS un modele 70B Les petits modeles necessitent des prompts radicalement differents. Chaque token compte. Privilegier la demonstration a l'instruction.
PrincipeExplicationExemple
COURT
<200 tok system		 Le system prompt du Narrator doit rester sous 200 tokens. Le GM sous 100. Au-dela, le modele perd le fil et la qualite chute. System: 150 tok au lieu de 400+ sur GPT-4
EXEMPLES > INSTRUCTIONS Un seul few-shot example vaut 10 lignes d'instructions. Le 1.5B imite tres bien mais comprend mal les meta-instructions abstraites. Montrer A) Traverser la foret plutot que «genere 3 choix avec des verbes d'action»
CONTRAINTES NEGATIVES Dire ce qu'il ne doit PAS faire est plus efficace que decrire le comportement souhaite. Le 1.5B respecte mieux les interdictions. JAMAIS de liste a puces. JAMAIS d'anglais. PAS de narration 1ere personne.
VOCABULAIRE INJECTE Injecter les mots exacts a utiliser dans le prompt. Le modele les reprendra naturellement dans sa generation. Vocabulaire: brume, dolmen, aubepine, triskel → apparaissent dans la sortie
STRUCTURE FIXE Imposer un format de sortie rigide avec delimiteurs clairs. Utiliser GBNF grammar quand possible. [NARRATION]...[CHOIX]A)...B)...C)...
PERSONA SIMPLE Definir le personnage en une phrase, pas un paragraphe. Ajouter 2-3 traits max. Tu es Merlin, druide FOU de Broceliande.

Prompt Narrator Optimise

System Prompt ~150 tokens

Tu es Merlin, druide FOU de Broceliande. Ton: {tone}. Ecris 2-3 phrases immersives en francais. Vocabulaire: {biome_words} + {celtic_words}. Puis 3 choix avec verbes PRECIS (A/B/C). JAMAIS de liste a puces. JAMAIS d'anglais. JAMAIS plus de 4 phrases de narration. Le choix C coute 1 Souffle. Format: [NARRATION] (2-3 phrases) [CHOIX] A) verbe + action B) verbe + action C) verbe + action (Souffle)

User Prompt ~100 tokens

Jour {day}/{max}. Saison: {season}. Biome: {biome}. Corps={state_corps}, Ame={state_ame}, Monde={state_monde}. Souffle={n}/{max}. {arc_context}. Dernier choix: {last_choice}.

Prompt Game Master Optimise

System Prompt ~80 tokens

Genere 3 effets JSON pour les choix A/B/C. Tout choix a un COUT (au moins un aspect baisse). Si danger (aspect Bas): une option DOIT soigner. Valeurs: -2, -1, 0, +1, +2. Somme totale par choix entre -1 et +1. Format strict JSON: {"A":{"corps":N,"ame":N,"monde":N},"B":{...},"C":{...}}

User Prompt ~60 tokens

Corps={val} ({state}) Ame={val} ({state}) Monde={val} ({state}). Souffle={n}/{max}. Scenario: {summary}. Choix A={label_a}. B={label_b}. C={label_c}.

Details des injections par situation

Joueur prudent — Detection

Active quand les 3 derniers choix du joueur sont tous A) ou B) (pas C/Souffle). Le code dans rag_manager.gd analyse choice_history et injecte le modificateur de ton.

# rag_manager.gd
func _detect_player_style() -> String:
    var recent = store.choice_history.slice(-3)
    var cautious = recent.filter(func(c): return c in ["A", "B"]).size()
    if cautious >= 3:
        return "Le voyageur est PRUDENT -- propose des choix subtils."
    elif recent.filter(func(c): return c == "C").size() >= 2:
        return "Le voyageur est TEMERAIRE -- propose des defis physiques."
    return ""

Arc CLIMAX — Detection

Active quand arc_progress >= 0.8 (80% de l'arc narratif). Le merlin_omniscient.gd calcule la progression et envoie le signal arc_climax_reached.

4e mur — Probabilite

Tire aleatoirement avec randf() < 0.067 (~1/15). Desactive si last_4th_wall_card est a moins de 5 cartes. Jamais pendant un arc CLIMAX.

Reve — Conditions

Active uniquement pendant la saison Samhain ou quand Ame == Possedee. Temperature elevee (0.85) et pas de contraintes de format strictes pour laisser le modele generer du contenu onirique.

[CHOIX] A) Se reposer sur la souche seche malgre le danger B) Avancer coute que coute vers la lumiere lointaine C) Implorer le Sanglier de preter sa force (Souffle)

GBNF Grammars (Annotated)

Qu'est-ce que GBNF ? GBNF (GGML BNF) est un format de grammaire supporte par llama.cpp et Ollama. Il contraint la sortie du LLM a respecter une structure syntaxique stricte. C'est essentiel pour le Game Master qui doit produire du JSON valide a 100%.
Narrator Output Grammar

Cette grammaire assure que le Narrator produit toujours le format [NARRATION]...[CHOIX]A)...B)...C)....

# Narrator GBNF Grammar v1.0 # Contraint la sortie a : narration (2-4 phrases) + 3 choix A/B/C root ::= narration choix # Narration block: 2-4 sentences in French narration ::= "[NARRATION]\n" sentence sentence sentence? sentence? "\n" # Each sentence: French text ending with punctuation sentence ::= [A-Z\xC0-\xFF] [^\n]+ [.!?] ( "\"" )? "\n" # Choices block: exactly 3 options choix ::= "[CHOIX]\n" option-a "\n" option-b "\n" option-c "\n"? # Each option starts with letter + ) and contains action text option-a ::= "A) " action-text option-b ::= "B) " action-text option-c ::= "C) " action-text souffle? # Action text: starts with a verb, one line action-text ::= [A-Z\xC0-\xFF] [^\n]+ # Optional Souffle cost marker souffle ::= " (Souffle)"
Limitations La grammaire Narrator est un guide, pas une contrainte dure. Ollama ne supporte GBNF que pour les modeles GGUF via llama.cpp. En pratique, on valide la sortie cote code avec le parser permissif de merlin_llm_adapter.gd.
GM Effects Grammar (Enforced)

Cette grammaire est strictement appliquee pour garantir du JSON valide. Le GM ne doit produire QUE du JSON.

# Game Master GBNF Grammar v1.0 # Contraint la sortie a un objet JSON avec exactement 3 cles A/B/C # Chaque cle contient corps, ame, monde avec valeurs entieres [-2, +2] root ::= "{" ws choices ws "}" choices ::= pair-a "," ws pair-b "," ws pair-c pair-a ::= "\"A\":" ws effect pair-b ::= "\"B\":" ws effect pair-c ::= "\"C\":" ws effect # Each effect has exactly 3 aspect modifiers effect ::= "{" ws "\"corps\":" ws val "," ws "\"ame\":" ws val "," ws "\"monde\":" ws val ws "}" # Allowed integer values: -2 to +2 # Note: accepts both "+1" and "1" formats val ::= "-2" | "-1" | "0" | "+1" | "+2" | "1" | "2" # Whitespace (optional) ws ::= [ \t\n]*

Fiabilite Avec la grammaire GBNF, le taux de JSON valide passe de ~85% (sans grammar) a ~99.5% (avec grammar). Les 0.5% restants sont geres par le JSON repair dans merlin_llm_adapter.gd.
